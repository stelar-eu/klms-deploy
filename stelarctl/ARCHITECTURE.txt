STELARCTL ARCHITECTURE
======================
Date: 2026-04-15


1. PRINCIPLES
--------------
stelarctl is the operational interface for STELAR deployment lifecycle:

  - deploy from a validated platform model
  - inspect live progress
  - purge target namespace resources

The first version intentionally stays simple:

  - no deployment state machine
  - no deploy-status annotation
  - no convergence timeout
  - no live platform model inference
  - no no-op detection
  - status is read-only and idempotent
  - deploy does not wait for readiness
  - hard redeploy is destructive and must be confirmed


2. COMMANDS
------------
  stelarctl deploy model.yaml [--env environments/<env>]
    Deploy or hard-redeploy using context and namespace from model.yaml.

  stelarctl status
    Show live progress in the active kubeconfig context and namespace.

  stelarctl teardown [--namespace <namespace>] [--env environments/<env>]
    Purge resources from the active kubeconfig context.

No command exposes --context.


3. TARGET RESOLUTION
---------------------
deploy:

  context   = model.k8s_context
  namespace = model.namespace

  The generated Tanka spec.json must use these exact values.
  tk apply must target these exact values.

status:

  context   = active kubeconfig context
  namespace = active kubeconfig namespace, or default when unset

teardown:

  context   = active kubeconfig context
  namespace = --namespace, otherwise active kubeconfig namespace, otherwise default


4. TANKA ENVIRONMENT
---------------------
A Tanka environment represents one deployment target:

  (model.k8s_context, model.namespace)

Default path:

  environments/<normalized-k8s-context>.<normalized-namespace>/

Example:

  model.k8s_context: minikube
  model.namespace: stelar

  -> environments/minikube.stelar/

Rules:

  - --env overrides the inferred path.
  - If env/spec.json exists, it must match model.k8s_context and model.namespace.
  - stelarctl owns these managed files:

      spec.json
      main.jsonnet
      model.yaml

  - On every deploy, stelarctl deletes those managed files if they exist.
  - stelarctl then regenerates spec.json and main.jsonnet.
  - stelarctl writes model.yaml only after tk apply succeeds.
  - stelarctl does not delete the environment directory.
  - stelarctl does not delete other files under the environment.


5. STATUS
----------
status is read-only. It must not create, patch, delete, or repair anything.

It returns DeploymentProgress:

  - active context
  - active namespace
  - active STELAR resources found: yes/no
  - progress percentage
  - progress bar
  - completed jobs / total required jobs
  - ready components / total required components
  - warnings

Progress:

    completed required jobs + ready required components
    -------------------------------------------------- * 100
       total required jobs + total required components

Interpretation:

  No active STELAR resources:
    No active deployment exists in the target.

  Active STELAR resources:
    A deployment exists. Progress is informational only in this version.


6. DEPLOY
----------
deploy is non-blocking with respect to readiness. It returns after tk apply
finishes.

deploy uses one flow for first deploy and hard redeploy. The only branch is
whether STELAR resources already exist in the target namespace.

Algorithm:

  1. Load and validate model.yaml.

  2. Resolve target from model.k8s_context and model.namespace.

  3. Resolve environment:

       --env provided -> use it
       --env omitted  -> infer from model target

  4. Validate existing env/spec.json against the model target.

  5. Run validate_model_against_cluster(model).

  6. Check whether labeled STELAR deployment resources exist in the model target.

  7. If no active STELAR resources exist:

       deploy fresh.

  8. If active STELAR resources exist:

       show current progress.
       warn that continuing will hard-redeploy the target.
       warn that namespace resources, including PVCs, will be purged.
       require confirmation before purging resources.

  9. If hard redeploy is confirmed:

       purge target namespace resources using the shared purge primitive.
       keep namespace.
       keep environment directory.

  10. Delete managed local environment files if they exist:

        spec.json
        main.jsonnet
        model.yaml

  11. Apply user and generated secrets.

  12. Generate spec.json and main.jsonnet.

  13. Run tk apply.

  14. On successful apply:

        write environment/model.yaml
        write namespace metadata annotations
        print current progress
        tell user to run stelarctl status

  15. If tk apply fails:

        do not write environment/model.yaml
        leave generated spec.json and main.jsonnet for diagnosis


7. HARD REDEPLOY AND TEARDOWN
------------------------------
Hard redeploy and teardown share:

  purge_resources_in_target(context, namespace)

teardown:

  purge resources from target namespace
  keep namespace
  keep environment directory

hard redeploy:

  purge resources from target namespace
  keep namespace
  keep environment directory
  then deploy again

teardown constraints:

  - context is always active kubeconfig context
  - --namespace overrides active kubeconfig namespace
  - --env is optional metadata/validation only
  - no --context
  - no --yes
  - no --delete-namespace
  - no --delete-env


8. ANNOTATIONS
---------------
Write only metadata:

  stelar.eu/tier
  stelar.eu/environment-id
  stelar.eu/environment-path
  stelar.eu/last-deployed-at
  stelar.eu/last-deployed-by
  stelar.eu/deploy-started-at

Optional future metadata:

  stelar.eu/deploy-owner
  stelar.eu/deploy-operation
  stelar.eu/model-hash
  stelar.eu/cluster-fingerprint

Do not store named states:

  deploy-status
  deploying
  ready
  failed
  degraded
  unknown


9. SECRETS
-----------
Secrets are applied from the input model during deploy.

Generated secrets are regenerated on every hard redeploy.

This version does not infer secret names or values from the cluster and does not
attempt no-op detection based on secrets.
